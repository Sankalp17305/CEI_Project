# Building GPT From Scratch: Character-Level Language Modeling on Shakespeare

This project builds a character-level **decoder-only Transformer** (a
miniature GPT) from scratch in PyTorch, trained on the **Tiny Shakespeare**
dataset (~1.1 million characters). It follows the progression from Andrej
Karpathy's "Let's build GPT" walkthrough: starting from the simplest
possible language model and building up, piece by piece, to a full
Transformer — so every component (self-attention, multi-head attention,
residual connections, layer norm) is understood rather than imported from
a library.

---

## 1. Project Structure

| File | Description |

| `bigram.py` | The starting-point baseline: a simple bigram language model |
| `gpt.py` | The final decoder-only Transformer (GPT) language model |
| `input.txt` | Training corpus — Tiny Shakespeare (~40,000 lines, ~1.1M characters) |

## 3. The Big Picture: What This Project Actually Builds Up To

The project (and the video it follows) doesn't jump straight to a
Transformer. It's built in deliberate stages, each one motivating the
next:

1. **Tokenize the raw text** at the character level.
2. **Establish a baseline** with the dumbest possible model (bigram) to
   see what "no real context modeling" looks like.
3. **Discover a mathematical trick**: how to efficiently average
   information from all previous positions using matrix multiplication
   instead of a loop.
4. **Turn that trick into self-attention** by making the averaging
   *learned* and *content-dependent* instead of uniform.
5. **Add multiple attention heads** so the model can track several kinds
   of relationships at once.
6. **Add a feed-forward layer** so each position gets extra computation,
   not just communication with other positions.
7. **Add residual connections and LayerNorm** so this can be stacked many
   layers deep without training instability.
8. **Assemble it all** into a full GPT, train it, and generate text.

Each stage below explains the "why," not just the "what."

## 4. Data Pipeline (shared by both scripts)

### 4.1 Tokenization

```python
chars = sorted(list(set(text)))
vocab_size = len(chars)
stoi = { ch:i for i,ch in enumerate(chars) }
itos = { i:ch for i,ch in enumerate(chars) }
encode = lambda s: [stoi[c] for c in s]
decode = lambda l: ''.join([itos[i] for i in l])
```

Every unique character in the corpus becomes one token, giving a small
vocabulary (`vocab_size` ≈ 65). This is the crudest possible tokenizer —
deliberately simple, unlike GPT's real subword (byte-pair encoding)
tokenizer with tens of thousands of tokens — so the project can focus on
the model architecture rather than tokenization.

### 4.2 Train/val split

```python
data = torch.tensor(encode(text), dtype=torch.long)
n = int(0.9*len(data))
train_data = data[:n]
val_data = data[n:]
```

A contiguous 90/10 split (no shuffling), since order matters for the
validation loss to be meaningful for a sequence model.

### 4.3 Batching (`get_batch`)

```python
ix = torch.randint(len(data) - block_size, (batch_size,))
x = torch.stack([data[i:i+block_size] for i in ix])
y = torch.stack([data[i+1:i+block_size+1] for i in ix])
```

Each training example is a `block_size`-length chunk (`x`) paired with the
same chunk shifted one character to the right (`y`) — the model learns
"predict the next character everywhere." A single chunk of length
`block_size` actually contains `block_size` separate training signals:
predict character 2 from character 1, character 3 from characters 1-2,
and so on up to the full window. This is the reason the model will later
need **causal masking** — so it can't cheat by looking at future
characters when learning to predict an earlier position.

### 4.4 Loss estimation (`estimate_loss`)

Periodically switches to `.eval()` mode, disables gradients, and averages
loss over `eval_iters` batches for both splits — giving a much smoother,
less noisy loss readout than a single training batch would.

## 5. Stage 1 — The Bigram Baseline (`bigram.py`)

```python
self.token_embedding_table = nn.Embedding(vocab_size, vocab_size)
logits = self.token_embedding_table(idx)  # (B, T, vocab_size)
```

The simplest possible model: given one character, directly look up a row
of logits — one score per possible next character. There is **no context
beyond the current single character** — predicting character 5 uses zero
information from characters 1–4.

This establishes the floor: the loss/quality you get with no sequence
modeling ability at all. It trains fast (`max_iters=3000`) and its
generated output is close to gibberish, since it can only capture
character-pair statistics (e.g. "q" is almost always followed by "u").
Everything from here on is motivated by fixing this limitation.

## 6. Stage 2 — The Mathematical Trick Behind Attention

Before the video calls anything "attention," it builds the core
mechanism in isolation: **how do you let each position average
information from all previous positions, efficiently?**

- **Naive version**: loop over each position, average the embeddings of
  everything before it. Correct, but slow (Python loop).
- **Matrix-multiply version**: build a lower-triangular matrix of ones,
  normalize each row to sum to 1, and multiply:
  ```python
  wei = torch.tril(torch.ones(T, T))
  wei = wei / wei.sum(1, keepdim=True)
  out = wei @ x
  ```
  This computes the exact same running average, but as a single batched
  matrix multiplication instead of a loop — the row-normalized
  lower-triangular matrix acts as a set of "attention weights" that are
  currently uniform (each past position weighted equally).
- **The key insight**: this uniform averaging is a special case of
  something more general. What if, instead of weighting every past
  position equally, the model *learned* how much to weight each one,
  based on the actual content at each position? That upgrade — from fixed
  uniform weights to learned, content-dependent weights — **is**
  self-attention.

---

## 7. Stage 3 — Self-Attention (the `Head` class in `gpt.py`)

```python
self.key   = nn.Linear(n_embd, head_size, bias=False)
self.query = nn.Linear(n_embd, head_size, bias=False)
self.value = nn.Linear(n_embd, head_size, bias=False)

k = self.key(x)
q = self.query(x)
wei = q @ k.transpose(-2,-1) * k.shape[-1]**-0.5   # (B, T, T)
wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
wei = F.softmax(wei, dim=-1)
v = self.value(x)
out = wei @ v
```

Every position produces three vectors via learned linear projections:
- **Query** — "what am I looking for?"
- **Key** — "what do I contain?"
- **Value** — "what do I actually pass along if attended to?"

The dot product of a position's query with every position's key gives an
**affinity score** — how relevant each other position is. These scores
are scaled by `1/sqrt(head_size)` to keep them from growing too large in
magnitude (which would push softmax into overly peaked, near one-hot
distributions and cause vanishing gradients).

**Causal masking** is what turns this into a *decoder*:

```python
self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size)))
wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
```

Positions above the diagonal (future tokens, relative to a given query
position) get masked to `-inf` before the softmax, forcing their
attention weight to zero. This means position `t` can only attend to
positions `0...t` — never anything after it. This is the fundamental
difference between an **encoder** (bidirectional attention, e.g. BERT)
and a **decoder** (causal attention, e.g. GPT): decoders must never see
the future, both because it would be cheating during training and
because during generation there *is* no future yet.

Finally, the (now content-dependent, non-uniform) attention weights
compute a weighted sum of the value vectors — same mechanical shape as
the earlier "trick," but the weights are now learned rather than fixed.

---

## 8. Stage 4 — Multi-Head Attention

```python
self.heads = nn.ModuleList([Head(head_size) for _ in range(num_heads)])
self.proj = nn.Linear(head_size * num_heads, n_embd)

out = torch.cat([h(x) for h in self.heads], dim=-1)
out = self.dropout(self.proj(out))
```

Running `num_heads` smaller attention heads in parallel — each with
`head_size = n_embd // n_head` — and concatenating their outputs lets the
model track several different kinds of relationships simultaneously
(e.g. one head might track nearby punctuation, another might track a
name mentioned several lines earlier). This is analogous to having
multiple independent "communication channels" between positions instead
of one large one of the same total size.

> **Note on the current implementation**: this version loops over heads
> with a Python `for` loop, which is correct but not the most efficient
> way to compute it — see [Section 12 (EX1)](#12-ex1-vectorizing-multi-head-attention)
> below for a vectorized version that computes all heads in a single
> batched operation instead of looping.

---

## 9. Stage 5 — Feed-Forward Network

```python
self.net = nn.Sequential(
    nn.Linear(n_embd, 4 * n_embd),
    nn.ReLU(),
    nn.Linear(4 * n_embd, n_embd),
    nn.Dropout(dropout),
)
```

Self-attention lets positions **communicate** with each other. The
feed-forward network gives each position extra **computation** on its
own, independent of the others — think of attention as "gathering
information from elsewhere" and feed-forward as "thinking about what was
gathered." The 4x expansion (`n_embd → 4*n_embd → n_embd`) mirrors the
original Transformer paper's design, giving the network more capacity to
compute nonlinear features before compressing back down.

## 10. Stage 6 — Residual Connections, LayerNorm, and the Transformer Block

```python
def forward(self, x):
    x = x + self.sa(self.ln1(x))      # attention sub-layer
    x = x + self.ffwd(self.ln2(x))    # feed-forward sub-layer
    return x
```

Two ingredients make it possible to stack many of these blocks (6, in
this case) without training collapsing:

- **Residual ("skip") connections** (`x = x + ...`): the sub-layer's
  output is *added* to its input rather than replacing it. This gives
  gradients a direct, unobstructed path back through every block during
  backpropagation — without this, deep stacks of layers suffer from
  vanishing gradients and become very hard to train.
- **Pre-norm LayerNorm** (`LayerNorm` applied *before* each sub-layer,
  not after): the modern convention (used from GPT-2 onward), more
  stable than the original 2017 Transformer's post-norm design, because
  the residual stream itself is never renormalized or interrupted — only
  the copy about to enter a sub-layer is.

Six of these blocks are stacked:

```python
self.blocks = nn.Sequential(*[Block(n_embd, n_head=n_head) for _ in range(n_layer)])
```

Each additional block lets the model compose progressively more abstract
features — early blocks might pick up local character patterns, later
blocks combine those into higher-level structure (word boundaries,
speaker-label formatting, sentence rhythm).

**Dropout** is layered throughout (attention weights, projections,
feed-forward) purely as a regularizer against overfitting, given the
model's size relative to the data.

## 11. Assembling the Full GPT

### 11.1 Hyperparameters

| Parameter | Value | Meaning |
|---|---|---|
| `batch_size` | 64 | sequences processed in parallel per training step |
| `block_size` | 256 | context length — how many previous characters the model attends to |
| `n_embd` | 384 | embedding / residual stream dimensionality |
| `n_head` | 6 | number of parallel attention heads per block |
| `n_layer` | 6 | number of stacked Transformer blocks |
| `dropout` | 0.2 | dropout probability |
| `learning_rate` | 3e-4 | AdamW learning rate |
| `max_iters` | 5000 | total training steps |

Total size: **~10.7M parameters**, printed at the start of training via
`sum(p.numel() for p in m.parameters())/1e6`.

### 11.2 Token + positional embeddings

```python
self.token_embedding_table = nn.Embedding(vocab_size, n_embd)
self.position_embedding_table = nn.Embedding(block_size, n_embd)
...
tok_emb = self.token_embedding_table(idx)
pos_emb = self.position_embedding_table(torch.arange(T, device=device))
x = tok_emb + pos_emb
```

Attention has no inherent sense of order, so the model needs both *what*
each character is (`tok_emb`) and *where* it sits in the sequence
(`pos_emb`, a learned embedding per position, added elementwise). Their
sum is the initial "residual stream" that flows through every block.

### 11.3 Output head

```python
self.ln_f = nn.LayerNorm(n_embd)
self.lm_head = nn.Linear(n_embd, vocab_size)
...
x = self.ln_f(x)
logits = self.lm_head(x)
```

After the last block, a final LayerNorm stabilizes the representation,
then a linear layer projects back up to `vocab_size`, producing a logit
for every possible next character at every position.

### 11.4 Weight initialization

```python
def _init_weights(self, module):
    if isinstance(module, nn.Linear):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
        if module.bias is not None:
            torch.nn.init.zeros_(module.bias)
    elif isinstance(module, nn.Embedding):
        torch.nn.init.normal_(module.weight, mean=0.0, std=0.02)
```

Weights are initialized from a narrow normal distribution (`std=0.02`,
matching GPT-2's scheme) instead of PyTorch's default, keeping
activations and gradients well-behaved early in training across several
stacked residual blocks.

### 11.5 Loss function

```python
B, T, C = logits.shape
logits = logits.view(B*T, C)
targets = targets.view(B*T)
loss = F.cross_entropy(logits, targets)
```

Every position in every sequence in the batch becomes one classification
example for a standard cross-entropy loss.

### 11.6 Training loop

```python
optimizer = torch.optim.AdamW(model.parameters(), lr=learning_rate)

for iter in range(max_iters):
    if iter % eval_interval == 0 or iter == max_iters - 1:
        losses = estimate_loss()
        print(f"step {iter}: train loss {losses['train']:.4f}, val loss {losses['val']:.4f}")
    xb, yb = get_batch('train')
    logits, loss = model(xb, yb)
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()
```

Standard PyTorch loop using AdamW (Adam with decoupled weight decay),
with periodic train/val loss logging so you can watch for overfitting
(val loss diverging upward while train loss keeps falling).

### 11.7 Autoregressive generation

```python
def generate(self, idx, max_new_tokens):
    for _ in range(max_new_tokens):
        idx_cond = idx[:, -block_size:]
        logits, loss = self(idx_cond)
        logits = logits[:, -1, :]
        probs = F.softmax(logits, dim=-1)
        idx_next = torch.multinomial(probs, num_samples=1)
        idx = torch.cat((idx, idx_next), dim=1)
    return idx
```

Text is generated one character at a time, feeding each newly sampled
character back in as input for the next step:
- **Cropping to `block_size`**: positional embeddings only exist for
  positions `0` to `block_size-1`, so the model can never see more than
  the last 256 characters — this is its fixed "context window."
- **Sampling via `torch.multinomial`**, not `argmax`: the next character
  is drawn probabilistically from the softmax distribution, giving
  natural variation rather than always picking the single most likely
  character.

## 12. EX1: Vectorizing Multi-Head Attention

A known follow-up exercise (from the same walkthrough this project is
based on) asks: combine `Head` and `MultiHeadAttention` into one class
that processes all heads in parallel, treating heads as an extra tensor
dimension rather than looping over them in Python.

**Why it matters**: the current `MultiHeadAttention` loops over
`num_heads` separate `Head` modules — correct, but inefficient, since
each head does identical operations with different weights. Real GPT
implementations (e.g. nanoGPT) do this in one batched operation instead.

**Vectorized implementation:**

```python
class MultiHeadAttention(nn.Module):
    """ multiple heads of self-attention, computed in parallel via batched tensor ops """

    def __init__(self, num_heads, head_size):
        super().__init__()
        self.num_heads = num_heads
        self.head_size = head_size

        # one big linear layer instead of num_heads separate small ones
        self.key   = nn.Linear(n_embd, n_embd, bias=False)
        self.query = nn.Linear(n_embd, n_embd, bias=False)
        self.value = nn.Linear(n_embd, n_embd, bias=False)

        self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size)))

        self.proj = nn.Linear(n_embd, n_embd)
        self.attn_dropout = nn.Dropout(dropout)
        self.proj_dropout = nn.Dropout(dropout)

    def forward(self, x):
        B, T, C = x.shape  # C == n_embd

        k = self.key(x)    # (B, T, C)
        q = self.query(x)  # (B, T, C)
        v = self.value(x)  # (B, T, C)

        # split C into (num_heads, head_size), then move heads next to batch dim
        k = k.view(B, T, self.num_heads, self.head_size).transpose(1, 2)  # (B, nh, T, hs)
        q = q.view(B, T, self.num_heads, self.head_size).transpose(1, 2)  # (B, nh, T, hs)
        v = v.view(B, T, self.num_heads, self.head_size).transpose(1, 2)  # (B, nh, T, hs)

        # attention scores for ALL heads at once — one batched matmul
        wei = q @ k.transpose(-2, -1) * self.head_size**-0.5   # (B, nh, T, T)
        wei = wei.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
        wei = F.softmax(wei, dim=-1)
        wei = self.attn_dropout(wei)

        out = wei @ v                           # (B, nh, T, hs)
        out = out.transpose(1, 2).contiguous()  # (B, T, nh, hs)
        out = out.view(B, T, C)                 # (B, T, C) -- heads concatenated back together

        out = self.proj_dropout(self.proj(out))
        return out
```

This one class replaces both the old `Head` and `MultiHeadAttention`
classes. Key mechanics:

1. **One linear layer instead of many.** Since `n_embd = num_heads *
   head_size`, a single `nn.Linear(n_embd, n_embd)` does the work of
   `num_heads` separate small linear layers in one matmul.
2. **`.view()` splits heads out**: `(B, T, C)` → `(B, T, num_heads,
   head_size)`, purely a reinterpretation of the same memory — no data
   movement.
3. **`.transpose(1, 2)` makes heads act like an extra batch dimension**:
   `(B, num_heads, T, head_size)`. PyTorch's batched matmul (`@`) treats
   leading dimensions as independent batches, so attention scores for
   *every* head and *every* batch element are computed in a single
   operation instead of a loop.
4. **The attention math itself is unchanged** — masking, softmax,
   weighted sum against `v` — just computed once for all heads instead
   of `num_heads` separate times.
5. **Recombining heads**: `.transpose(1, 2)` then `.view(B, T, C)` merges
   heads back into one `n_embd`-sized dimension, equivalent to the
   original `torch.cat([...], dim=-1)`.
6. **`.contiguous()`** is required before the final `.view()`, since
   `.transpose()` only changes stride metadata, not actual memory layout
   — `.view()` needs contiguous memory to safely reinterpret shape.

With matching weights, this produces numerically identical results to
the original loop-based version — the benefit is purely computational
efficiency (fewer, larger matmuls run more efficiently than many small
ones in a Python loop, especially on GPU).

## 13. Bigram vs. GPT — What Changed and Why It Matters

| Aspect | Bigram | GPT |
|---|---|---|
| Context used per prediction | 1 previous character | up to 256 previous characters |
| Positional awareness | None | Learned positional embeddings |
| Parameters | `vocab_size²` (tiny) | ~10.7M |
| Core mechanism | Embedding lookup | Self-attention + MLP, stacked 6x |
| Long-range dependency modeling | No | Yes, within the context window |

The bigram model can only ever learn character-pair frequencies. The GPT
model, through stacked self-attention, conditions each prediction on the
full preceding context (up to 256 characters) — which is why its output
captures Shakespeare's *form* (character-name headers, verse-like
structure, archaic phrasing) far better, even though neither model has
any real semantic understanding of the text.
